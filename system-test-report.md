## Bug / 风险点

### B1. 任务详情页在登出或切换账号时可能触发前端运行时错误

严重程度：中  
影响范围：`/tasks/[id]` 学生端任务详情页  

复现现象：

1. 学生登录并进入活动详情页。
2. 切换账号或退出登录。
3. 浏览器控制台出现：

```text
TypeError: Cannot read properties of null (reading 'id')
```

本轮测试未看到页面直接白屏，但这是 React 运行时错误，说明组件在用户状态为空时仍读取了用户 ID。

可能原因：

- `app/tasks/[id]/page.tsx` 中向 `SubmitTab` 和 `LeaderboardTab` 传入了 `currentUserId={user!.id}`。
- 当 `user` 因登出、切换账号或 auth hydration 短暂为 `null` 时，仍可能用到旧的 `task/enrollment` 状态继续渲染，从而触发 `user!.id`。

建议：

- 在渲染任务详情主体前显式要求 `user` 存在，例如 `if (loading || fetching || !task || !user) return ...`。
- 或者把 `currentUserId` 改成可空值，并让子组件处理未登录 / 未就绪状态。
- 登出时也可以清空任务页相关状态，避免旧状态参与下一轮渲染。

### B2. 管理员评分器删除缺少确认，也没有处理删除失败

严重程度：中  
影响范围：`/admin/judge-profiles`  

发现方式：源码核对 + UI 风险检查。

现状：

- 点击评分器“删除”会直接发送 DELETE，没有二次确认。
- 前端没有检查 `res.ok`，请求失败也会刷新并提示“已删除”。
- 如果评分器已被活动引用，后端可能因为外键约束失败返回 500，但前端仍可能给出成功提示。

涉及位置：

- `app/admin/judge-profiles/page.tsx`
- `app/api/admin/judge-profiles/[id]/route.ts`

建议：

- 删除前增加确认弹窗，明确“正在使用的评分器不能删除”。
- 前端检查响应状态，失败时显示后端错误。
- 后端删除前检查是否有 Task 引用，返回 400 和可读错误，例如“该评分器已被活动使用，无法删除”。

### B3. 管理员 LLM 配置删除缺少确认，后端外键错误提示不够友好

严重程度：中  
影响范围：`/admin/llm-configs`  

现状：

- 点击 LLM 配置“删除”会直接发送 DELETE，没有二次确认。
- 前端会在失败时提示“删除失败（可能有评分器正在使用）”，但后端没有显式检查引用关系，实际可能是数据库约束错误。

建议：

- 删除前增加确认弹窗。
- 后端删除前检查评分器引用，返回明确错误。
- 如果该 LLM 配置已被评分器使用，在列表中直接禁用删除按钮或显示“被 N 个评分器使用”。

## 可优化体验

### O1. `/admin` 直接访问会跳回活动广场，不像管理控制台首页

现状：

- 直接访问 `/admin` 会 redirect 到 `/dashboard`。
- 但导航栏有“管理控制台”的概念，用户可能期待 `/admin` 是一个管理首页。

建议：

- 提供一个简单的管理控制台首页，放用户管理、审批、LLM 配置、评分器、题库、系统设置的快捷入口。
- 或者减少 `/admin` 暴露，避免用户以为有单独后台首页。

### O2. 移动端表格可读性一般

现状：

- 移动端可以渲染活动列表和任务详情，但表格列较多，尤其管理员活动列表、用户管理、题目表、提交表在窄屏下阅读压力较大。

建议：

- 移动端把表格改为卡片式列表，或给表格容器增加明确的横向滚动提示。
- 常用操作按钮在移动端可以收进“更多”菜单。

### O3. 删除/重置类高风险操作交互不统一

现状：

- 有些页面使用浏览器原生 `confirm`。
- 有些删除没有确认。
- 有些失败提示较泛。

建议：

- 统一使用项目内 Dialog 组件。
- 高风险操作文案包含影响范围，例如“会删除报名、提交、作答记录”。
- 删除失败时显示具体原因，例如“已有活动引用，无法删除”。

### O4. 学生端“答题”页承担多个任务，命名可以更明确

现状：

- “答题”页同时包含 Prompt 保存、公开题试跑、正式提交和提交记录。
- 对第一次使用的学生来说，“答题”不一定能暗示“正式提交评测”也在这里。

建议：

- Tab 名称可改为“答题/提交”或“提交评测”。
- 在页面顶部增加一行简短状态说明：先试跑公开题，确认后点击提交评测。

### O5. 智能体配置页可以补充接入方式示例

现状：

- OpenAI 兼容 API、Dify、Coze 都支持，但学生需要知道 endpoint 应该填哪一层地址。

建议：

- 在切换接入方式时显示 1 行示例：
  - OpenAI 兼容：`https://example.com/v1`
  - Dify：应用 API Endpoint，不包含查询参数
  - Coze：Bot ID、Endpoint 和 API Key 分别从哪里获取
- 连通性失败时在错误信息下方给出常见排查项。

## 后续建议

优先级建议：

1. 先修 B1，避免任务页在登出 / 切换账号时产生运行时错误。
2. 再修 B2/B3，避免管理端误删或错误提示不准确。
3. 然后做 O1，补上管理控制台首页。
4. 最后逐步优化移动端表格和学生端说明文案。
