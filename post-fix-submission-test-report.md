## 可优化体验

### O1. 智能体大赛默认提示不足，学生模型容易误解题目

现象：
- 智能体大赛真实提交完成，但公开分为 0。
- 抽查答案发现模型把 `7,12,12,13` 这类题目理解成“数列规律/预测下一个数”，而不是“24 点游戏”。
- 这不是系统调用失败，生成和评分链路都正常；更像是学生侧默认提示不足。

建议：
- 在智能体大赛的学生配置页增加“推荐 System Prompt”或“一键填入 24 点提示词”。
- 或者活动发布者可以在活动级别配置一段“题目说明/参赛提示”，学生进入 Chatbot 配置和提交页时可见。

### O2. 提交前连通性测试会额外消耗一次模型调用

现象：
- 当前提交流程会先调用 `/enrollment/validate`，成功后再创建正式提交。
- 这能提前发现配置错误，但每次正式提交都会额外调用一次模型。

建议：
- 保留安全检查，但可以缓存最近一次成功连通性测试，例如 5 分钟内配置未变化则不重复探测。
- 或在提交按钮旁提示“提交前会先做一次连通性检查”。

### O3. 刷新页面后运行中提交只有列表轮询，没有恢复进度条

现象：
- 提交创建后页面会展示进度条。
- 如果学生在提交运行中刷新页面，提交记录会轮询更新 `PENDING/RUNNING/COMPLETED`，但不会自动重新连接该提交的 SSE 进度流。

建议：
- 页面加载提交列表时，如果发现当前用户有 `PENDING` 或 `RUNNING` 提交，可以自动调用 `startProgress(submission.id)` 恢复进度条。

### O4. 智能体大赛活动里残留管理员统一 LLM 字段，容易造成数据层理解歧义

现象：
- `测试-智能体大赛说明书截图` 的 `adminLLMEnabled=false`，实际不会使用管理员统一 LLM。
- 但数据库中仍残留 `adminModel` 和 `adminStudentLLMConfigId`。

影响：
- 当前业务逻辑按 `adminLLMEnabled` 判断，因此不影响提交。
- 但后续导出配置、复制活动或排查问题时，容易误以为智能体活动仍使用管理员 LLM。

建议：
- 当活动关闭管理员统一 LLM 时，保存时清空或隐藏这些字段。
- 克隆活动时也可以根据 `adminLLMEnabled` 做一次字段规整。
